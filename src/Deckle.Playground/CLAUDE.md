# CLAUDE.md — Deckle.Playground

Surface de tuning développeur de l'app. Héberge la `PlaygroundWindow` (NavigationView Auto + Frame de pages), trois pages owned (`HomePage` landing, `HudPage` pour la composition stroke, `AmbientPage` pour la capture écran + Hue + tuning HDR), un `PlaygroundShell` static qui sert de point de routage navigation côté pages, et deux ViewModels (`HudViewModel`, `AmbientViewModel`). Le module est dev-only : il existe pour permettre à Louis d'ajuster en live les paramètres des pipelines tournants dans l'app sans rebuild, et pour servir d'antichambre aux refontes UX avant qu'elles ne migrent dans Settings ou dans une autre surface utilisateur finale.

L'architecture détaillée (shell + 3 pages, pattern Tag → Type.GetType, MVVM partiel ciblé sur ce qui paie, lifetime fenêtre Hide-on-close + dispose ressources page au Closed terminal) suit exactement le pattern de `Deckle.Settings` — lire `src/Deckle.Settings/CLAUDE.md` avant de toucher au shell ou au routage. La doctrine Settings (Auto-save partout, NavigationView Auto, SettingsCard / SettingsExpander, header H1 par page) ne s'applique pas verbatim ici parce que le Playground n'est pas une page de réglages utilisateur final — c'est un atelier de tuning avec des sliders denses, des previews live et des panneaux générés programmatiquement.

## Patterns non négociables

**Pas de persistance HUD.** Toute valeur de tuning manipulée dans `HudPage` vit en mémoire pour la durée du processus et meurt à l'app exit. Le bouton « Reset all » et les boutons Reset par section snap aux defaults compilés. Le but du Playground HUD est de trouver les bons defaults, pas de les mémoriser — un réglage qui se révèle juste migre dans le code comme nouveau default, le Playground ne stocke rien. Si une persistance par session devient nécessaire, c'est un nouveau service dédié (`PlaygroundSettingsService`), jamais une dérive vers `AmbientSettingsService` ou un fichier ad-hoc.

**Persistance Ambient via le service partagé.** `AmbientPage` lit et écrit `AmbientSettingsService.Instance.Current` (fichier `<UserDataRoot>/modules/ambient/settings.json`). Les knobs tunés dans le Playground sont la même source de vérité que ceux de la page Settings Ambient — toute modification s'applique live au moteur `AmbientEngine.Current` qui tourne dans l'app hôte. Le pattern de propagation va dans les deux sens : le Playground observe `AmbientSettingsService.Changed` pour refléter les modifications faites depuis Settings, et la page Settings fait de même pour celles faites depuis le Playground.

**MVVM ciblé, pas mécanique.** `HudViewModel` se limite à l'état transport et à la cible courante (`CurrentTarget`, `IsPlaying`, propriétés calculées `IsPlayEnabled` / `IsPauseEnabled` / `IsStopEnabled`, `CurrentTargetLabel`). `TuningModel` reste un POCO mutable à champs publics directement manipulé par les lambdas des sliders dans le code-behind de `HudPage` — wrappage `[ObservableProperty]` rejeté parce que le panneau de tuning est construit programmatiquement sans `x:Bind`, donc la duplication n'achète aucune valeur de binding. `AmbientViewModel` en revanche suit le pattern complet de `GeneralViewModel` (partial properties, flag `_isSyncing`, `Load()` / `PushToSettings()` via les `partial void OnXxxChanged`) parce que les sliders touchent un store persisté et que la centralisation des writes + side-effects (flip vers `AmbientMode.Custom` à chaque tuning) rend le pattern rentable.

**Toolbar HUD : DropDownButton groupé + transport enable-bound.** Le picker de target est un `DropDownButton` à `MenuFlyout` avec deux `MenuFlyoutSubItem` ("States" pour Charging/Recording/Transcribing/Rewriting, "Primitives" pour Conic/ArcMask/Combined). Le label du bouton reflète la cible courante via `{x:Bind ViewModel.CurrentTargetLabel}`. Le transport est trois boutons explicites Play / Pause / Stop dont l'`IsEnabled` est binded aux propriétés calculées du VM — pas de toggle qui swap son icône sous le curseur, l'utilisateur voit en permanence quelles actions sont possibles. Le panneau de tuning ne montre que les `SettingsExpander` pertinents pour la cible active (mappage défini par `HudViewModel.ActiveTuningSections`), avec l'expander Parked toujours appendu en bas (collapsed par défaut) pour les valeurs de transition rarement touchées.

**Page contract aligné Settings.** Chaque page hérite de `Page`, déclare `NavigationCacheMode = NavigationCacheMode.Required` dans le constructeur, et survit aux navigations dans le `Frame`. Les ressources runtime (composition naked preview, screen capture service, frame sampler, Hue REST output, preview cells) vivent comme champs d'instance de la page et sont disposées via `DisposeResources()` appelé par `PlaygroundWindow.Closed` (terminal, pas Closing→Hide qui doit préserver l'état). Le `OnNavigatedTo` recharge le ViewModel depuis le service de settings ; le `OnNavigatingFrom` arrête les timers UI off-screen sans toucher au pipeline canonique qui continue de pousser vers Hue.

## PlaygroundShell — registry de delegates

Miroir de `SettingsHost`. Une classe statique avec un seul delegate aujourd'hui (`NavigateTo` invoqué par `HomePage` pour basculer la sélection NavView vers HUD ou Ambient). Le pattern évite à `HomePage` de tenir une référence directe à `PlaygroundWindow`. Si d'autres callbacks émergent, ils s'ajoutent comme champs statiques typés nommés — pas de dictionnaire de strings, pas de Service Locator déguisé. Le shell pose le delegate en constructeur et le retire au `Closed` pour qu'une page survivante ne route pas vers une fenêtre détruite.

## Lifetime fenêtre

Singleton lazy créé par `App.OnLaunched` au premier `ShowPlaygroundLazy()`, jamais détruit avant `QuitApp`. `Closing` cancel + Hide via `AppWindow.Hide()` (le pattern Win32 `SW_HIDE` utilisé par Settings n'est pas reproduit ici parce que Playground n'a pas le même symptôme DComp swap chain ; à ré-évaluer si le bug remonte sur Playground). `Closed` n'arrive qu'au quit terminal et c'est là que les `DisposeResources()` des pages sont appelés pour libérer la composition Win2D, le screen capture service, le frame sampler, et désabonner les observateurs `HuePairingService.BridgeChanged` + `AmbientSettingsService.Changed`.

## Limites assumées

Pas de localization complète : les strings denses dans les SettingsExpander, les labels de slider, les captions des cards, les textes RadioButtons sont hard-codés en anglais. Seuls les titres haut niveau (TitleBar, NavView items, H1 de page, headers et descriptions des cards Home) passent par `x:Uid` + `Strings/en-US/Resources.resw`. Le backlog localization du projet a explicitement laissé Playground hors V0.1 — la migration intégrale viendra dans une passe dédiée alignée sur les autres surfaces. Les icônes glyph FontIcon sont également hard-codées via leur code-point Segoe Fluent Icons et appellent une centralisation future via un ResourceDictionary glyph keys (idée mémoire projet, pas encore réalisée).

Pas de tests automatisés sur ce module : le Playground étant une surface dev et son rendu étant pour l'essentiel visuel + interactif, la vérification reste manuelle pour l'instant. La refonte logging EventSource + tests à venir cible d'abord les modules production critiques (`Deckle.Audio`, `Deckle.Whisp`, `Deckle.Lighting.Ambient` pour les pushes Hue) ; le Playground viendra si une régression visible le motive.
